# Drive / Studio 无数据状态方案评审结论

## 1. 评审范围

本次评审针对 `drive-studio-empty-state-technical-design.md` 中提出的三个候选方案，重点审查：

- 是否满足需求；
- 是否符合现有架构；
- 是否控制改动范围；
- 是否便于后续维护。

## 2. 评审结论

最终确认采用：

- **方案 C：抽取轻量导航工具并复用**

## 3. 评审理由

### 满足需求最完整

Drive 空状态会直接修正为打开 Control；
Studio 空状态继续保持打开 Control，不会因重构而退化。

### 与现有架构最一致

Background 已经承担 Control / Drive 的窗口调度职责；
继续复用 `OPEN_CONTROL_WINDOW` / `OPEN_DRIVE`，比直接写死 URL 更符合当前工程设计。

### 改动小且边界清晰

只涉及：

- 一个轻量导航工具；
- Drive 空状态按钮交互；
- Studio 空状态导航调用点。

不会触碰录制状态机、Worker、OPFS 和导出链路。

### 可维护性更好

后续如果 Control 或 Drive 的开窗方式需要调整，只需修改一个工具文件，不需要在多个页面重复同步。

## 4. 实施要求

1. 保持 Drive 和 Studio 的空状态视觉结构基本不变；
2. 只修正录制引导行为，不扩大改动范围；
3. 保留 fallback，确保在非扩展上下文也能打开目标页面；
4. 实施后完成构建校验和截图验证。
