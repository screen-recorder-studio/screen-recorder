# Action 入口改造与卸载率治理评估

## 1. 文档信息

- 项目：Screen Recorder Studio
- 评估时间：2026-03-07
- 评估对象：
  1. Action 下拉菜单：录制 / Drive / Studio
  2. 点击 Studio 时直接打开 Studio，并读取 Drive 中第一条视频
  3. Studio 无视频时增加录制引导
  4. Studio 内弹出 Drive，复用 Drive 组件
- 结论摘要：
  - **这些改进有较大概率降低“入口认知错误”和“历史内容不可达”导致的流失与卸载，但不能单独视为卸载率问题的全部解法。**
  - **优先级最高的是：Action 菜单重构 + Studio 空状态引导。**
  - **Studio 直开第一条视频是有效改进，但必须定义“第一条”的业务语义，并补上无数据/坏数据兜底。**
  - **Studio 内嵌 Drive 面板能提升回看与二次编辑效率，但价值主要体现在复访留存和导出效率，不是首因级的卸载率治理手段。**

---

## 2. 背景与问题定义

当前版本的主要入口问题是：

1. 用户点击扩展 action 后，系统**直接进入录制流程入口**；
2. 历史录制内容被放在 Drive 中，但用户需要先进入录制面板，再点击 Drive，路径较长；
3. 对已经录过视频、希望“继续处理历史内容”的用户而言，主入口与用户目标不匹配；
4. Studio 作为高价值页面，没有成为 action 一级入口，导致“录制完成后的编辑能力”曝光不足。

这类问题并不一定直接表现为“用户说功能不好”，更常见的结果是：

- 第一次点击后没有看到自己预期的内容，形成“这个扩展只能录，不能管理”的认知；
- 回访用户找不到历史视频，误以为录制没有保存或产品不稳定；
- 用户完成一次录制后，第二次回来时找不到上次结果，降低再次使用意愿；
- 用户的核心任务是“管理 / 查看 / 导出已有录制”，却被强迫走“先录制”的路径，产生明显摩擦。

**因此，这不是单纯的交互细节问题，而是信息架构问题。**

---

## 3. 当前实现现状评估

基于当前工程实现，可得出以下现状：

### 3.1 Action 当前行为

- `background.ts` 中监听 `chrome.action.onClicked`；
- 点击 action 后，直接弹出 `control.html` 对应的独立 popup window；
- 若该窗口已存在，则只做聚焦恢复；
- 当前 manifest 未配置 `default_popup`，因此 action 入口不是原生下拉 popup，而是后台主动开窗。

**影响：**
- 对“马上开始录制”的用户是直达；
- 对“查看历史 / 编辑历史”的用户则是误导型入口。

### 3.2 当前 Drive 与 Studio 的关系

- `Drive` 页面负责遍历 OPFS 中 `rec_*` 目录并生成录制列表；
- `RecordingCard` 内可打开 `studio.html?id=...`；
- `Studio` 当前依赖 URL 参数 `id` 才能加载指定录制；
- `background.ts` 在收到 `OPFS_RECORDING_READY` 后会自动打开对应 `studio.html?id=...`。

**影响：**
- 录制完成后，Studio 已被系统自动打开，链路对“刚录完”的用户是成立的；
- 但对“回访用户”而言，Studio 缺少“无参数时的默认载入逻辑”和“内容缺失时的引导”。

### 3.3 当前空状态能力

- Drive 页面已有空状态；
- Studio 页面本身没有建立成体系的“无视频引导页”；
- `VideoPreviewComposite` 在 `!encodedChunks.length` 时只是停止处理，没有形成面向用户的明确解释。

**影响：**
- Studio 作为一级入口的前提条件尚未具备；
- 如果直接让用户从 action 点击 Studio，很容易出现“空白 / 不知道发生什么”的感受。

### 3.4 当前组件复用基础

- Drive 已拆出 `RecordingList.svelte` 和 `RecordingCard.svelte`；
- 但它们仍偏向 Drive 页面上下文，存在页面行为耦合；
- 从工程角度看，**具备复用基础，但不是零成本复用**。

## 4. “是否能降低卸载率”的核心判断

## 4.1 可以降低，但属于“中高影响、非唯一因素”

我认为本次改造**能降低部分卸载率**，主要作用于以下几类流失：

### A. 入口预期不匹配导致的早期流失

用户点击扩展图标时，真实意图可能是：

- 我要开始录制；
- 我要找到上次录制；
- 我要继续编辑、导出；
- 我要确认视频到底有没有保存成功。

当前产品只支持第一种意图的直达，其余三种都被隐藏在二级路径中。  
这种“入口只服务单一意图”的设计，通常会带来：

- 首次使用后误判产品能力边界；
- 二次回访时找不到资产；
- 认为产品只会“录一下”，缺少“管理”和“编辑”能力。

将 Action 入口升级为三分流菜单，能显著减少这一类误判。

### B. 资产不可见导致的信任流失

录屏类产品的留存基础，不只是“能录”，而是：

- 能不能找到录过的内容；
- 能不能继续处理；
- 能不能让我确信保存成功。

当前历史录制不在 action 一级入口暴露，会弱化“资产存在感”。  
而 Drive / Studio 成为一级入口后，用户更容易建立“录制结果是可持续管理的数字资产”的认知。

### C. 回访型用户的任务成本过高

对于复访用户，常见目标不是再次录制，而是：

- 回看最近录制；
- 补导出；
- 修改背景、裁剪、重新输出；
- 删除无用内容。

如果每次都要先打开录制控制页，再转跳 Drive 或 Studio，动作成本过高。  
这种问题会直接影响**次日 / 次周回访体验**，属于典型的“功能存在但入口失败”。

---

## 5. 对四项改进的逐项评估

## 5.1 改进一：增加 Action 下拉菜单（录制 / Drive / Studio）

### 价值判断

**价值：高**  
**对卸载率影响：高**  
**建议优先级：P0**

### 为什么有效

1. **把单一路径改成三意图入口**
   - 录制：服务首次和高频录制用户；
   - Drive：服务“找内容 / 管内容”的用户；
   - Studio：服务“编辑 / 导出”的用户。

2. **提升功能可发现性**
   - 当前很多能力已经存在，但不在 action 一级曝光；
   - 一级曝光本身就能提升认知完整度。

3. **降低“原来没有历史记录 / 编辑能力”的误解**
   - 这类误解往往不会转化为反馈，而是直接卸载。

### 风险与注意点

1. **不能把 Action 菜单做成新的复杂页面**
   - Action 菜单的本质应该是“任务分流器”，不是第二个控制台。

2. **“录制”按钮仍需足够强**
   - 不能因为增加入口而削弱主路径；
   - 录制仍应保持最高优先级和最快路径。

3. **Chrome 扩展原生能力限制**
   - 需求写的是“下拉菜单”，但从当前架构看，更适合落成 **action popup launcher**，而不是后台主动打开小窗模拟菜单；
   - 若继续沿用 `chrome.action.onClicked + windows.create` 模式做“伪菜单”，一致性与体验都不如原生 popup。

### 评估建议

**建议采用“原生 Action Popup 作为 Launcher”的方案，而不是继续用后台开窗模拟菜单。**

原因：

- 更符合 Chrome 扩展用户心智；
- 交互更快；
- 实现更稳定；
- 更适合作为轻量分流入口。

---

## 5.2 改进二：点击 Studio 时直接弹出 Studio，读取 Drive 中第一条视频

### 价值判断

**价值：高**  
**对卸载率影响：中高**  
**建议优先级：P0**

### 为什么有效

1. **把 Studio 从“结果页”升级为“可直接进入的工作台”**
   - 用户不一定是刚录完才需要 Studio；
   - 很多回访用户是为了继续处理旧视频。

2. **避免用户点进 Studio 后再做一次列表选择**
   - 对“我就想继续上次工作”的用户，直接打开最近一条内容更高效。

3. **可以强化“这个产品不是一次性录屏工具，而是录制工作台”的认知**

### 关键风险

1. **“第一条视频”语义不清**
   - 是 Drive 列表第一条？
   - 是创建时间最新一条？
   - 还是最近修改一条？

2. **如果没有视频，会导致 Studio 入口失效**
   - 这会让本次改造失分；
   - 所以 Studio 空状态不是增强项，而是前置条件。

3. **如果第一条视频损坏 / 不完整，会把失败体验放大**
   - 需要从“第一条任意记录”升级为“第一条可用记录”。

### 评估建议

**建议把“第一条视频”定义为：Drive 默认排序下的第一条可用录制，也就是“最新一条可正常打开的录制”。**

不建议简单按目录扫描顺序取第一条，原因是：

- 与用户在 Drive 中看到的顺序不一致；
- 可能读到损坏或旧版本残留目录；
- 容易造成“为什么不是我刚录的那个”的困惑。

### 结论

这个改造是对的，但必须同时满足三条：

1. 有默认载入规则；
2. 有“无录制可打开”时的空状态；
3. 有“坏录制跳过 / 降级”的兜底逻辑。

---

## 5.3 改进三：Studio 中优化，没有视频时给出录制引导

### 价值判断

**价值：极高**  
**对卸载率影响：高**  
**建议优先级：P0，且必须与改进二绑定**

### 为什么这是必做项

如果 Studio 被提升为一级入口，但没有空状态设计，那么用户在以下场景会直接看到失败体验：

- 首次安装后从未录制过；
- 用户清空了 Drive；
- OPFS 不可访问；
- 最近录制损坏或读取失败；
- 用户从 action 直接点了 Studio，但没有 `id` 参数。

这类失败不是“功能出错”，而是“入口无解释”，最容易造成：

- 不理解产品；
- 怀疑产品不稳定；
- 当场离开甚至卸载。

### 应承担的产品职责

Studio 的空状态不应只是“暂无视频”，而应承担以下职责：

1. 告知当前为什么没有内容；
2. 告知录制后会发生什么；
3. 提供可直接执行的下一步；
4. 提供回到 Drive 的查看入口；
5. 说明 Studio 的核心价值，让用户知道来这里是干什么的。

### 评估建议

Studio 空状态建议至少包含：

- 标题：还没有可编辑的视频
- 说明：完成录制后，视频会自动出现在这里
- 主按钮：开始录制
- 次按钮：打开 Drive
- 辅助说明：支持预览、裁剪、背景、美化、导出

### 结论

**如果只做“Studio 直开第一条视频”，不做空状态，本次改造风险较高。**

---

## 5.4 改进四：Studio 中的 Drive 改为在本页面弹出，复用 Drive 中组件

### 价值判断

**价值：中高**  
**对卸载率影响：中**  
**对复访留存与编辑效率影响：高**  
**建议优先级：P1**

### 为什么有效

1. **减少页面切换**
   - 用户在 Studio 中切换其他录制，不必再开新页面或新标签。

2. **提升“编辑工作台”完整度**
   - Studio 不再只是单视频编辑页，而是带资产切换能力的工作台。

3. **增强连续任务体验**
   - 查看 A → 切 B → 导出 → 删除 → 再切回，流程更顺。

### 为什么不是首要卸载率抓手

因为它优化的是**已进入 Studio 的用户体验**。  
而高卸载率当前更像是发生在“入口阶段”和“第一次找不到资产阶段”。

换句话说：

- Action 菜单和 Studio 空状态，解决的是**进不来 / 看不懂**；
- Studio 内嵌 Drive，解决的是**来了之后用得更顺**。

### 工程与产品风险

1. Drive 组件虽可复用，但仍有页面耦合；
2. 若把完整 Drive 直接塞进 Studio 右侧，信息密度会过高；
3. 若设计不当，会破坏当前 Studio 的编辑专注性。

### 评估建议

**建议采用“Studio 内侧边抽屉 / Modal Sheet”的方式嵌入 Drive 列表，而不是把 Drive 永久并排展开。**

理由：

- 更符合“需要时再打开”的使用频次；
- 不会压缩主预览区；
- 复用现有列表组件的成本更可控。

## 6. 综合优先级建议

## 6.1 推荐实施顺序

### P0：必须先做

1. Action 三入口菜单
2. Studio 默认直达最近一条可用录制
3. Studio 空状态与录制引导

### P1：紧随其后

4. Studio 内嵌 Drive 面板（抽屉 / 弹层）

### P2：后续增强

5. 在 Action 菜单中显示“最近一次录制时间 / 最近视频标题”
6. 在 Studio 空状态中增加“示例视频 / 使用说明”
7. 在 Drive / Studio 间增加最近打开记录

---

## 7. 对“卸载率”改善幅度的预估

以下为产品层面的相对判断，不是统计学承诺值：

### 7.1 预期能改善的指标

1. **首周卸载率**
   - 原因：减少“点进去不是我想要的”造成的失望退出。

2. **二次访问率 / 复访率**
   - 原因：用户更容易找回历史视频并继续编辑。

3. **Drive / Studio 使用率**
   - 原因：一级入口曝光。

4. **录制完成后的编辑转化率**
   - 原因：Studio 心智前置。

### 7.2 改善幅度判断

- **Action 菜单**：对卸载率有最直接影响，预期为中高改善；
- **Studio 默认打开最近录制**：对复访体验改善明显，预期为中等改善；
- **Studio 空状态**：对首访失败体验改善明显，预期为中高改善；
- **Studio 内嵌 Drive**：对重度用户和复访用户改善明显，预期为中等改善。

### 7.3 不能指望它解决的部分

如果卸载率同时受以下因素影响，本次改造无法单独解决：

- 首录权限申请失败；
- 录制质量或稳定性问题；
- 导出过慢；
- 语言与本地化不完善；
- 新手不理解 tab/window/screen 区别；
- 录制结束后“保存在哪”的认知不清。

因此，本次改造应被定义为：

> **“入口架构与资产可见性治理”专项，而不是整个卸载率问题的唯一解。”**

---

## 8. 关键产品建议

## 8.1 建议的一句话产品策略

**把 action 从“直接开始录制的按钮”升级为“录制工作台入口”。**

### 原因

录屏产品不是只有录制动作，还包括：

- 启动录制；
- 管理录制；
- 继续编辑；
- 导出成品。

当前入口只承接了第一件事，导致产品价值面被严重低估。

---

## 8.2 最推荐的目标体验

用户点击扩展图标后，首先看到一个轻量 launcher：

- **录制**：打开 Control，立即开始录制准备
- **Drive**：直接打开录制库
- **Studio**：直接进入最近一条可用视频；若没有则显示引导页

这样能让首次用户和回访用户都快速找到自己的路径。

---

## 8.3 对“Studio 读取第一条视频”的最终建议

建议不要在文档和产品文案里使用“第一条视频”这种易歧义表达，统一改成：

> **最近一条录制**

业务定义：

- 按 Drive 默认排序规则（创建时间倒序）取第一条；
- 跳过损坏 / 不完整 / 无法读取的数据；
- 若不存在可用视频，则进入 Studio 空状态。

---

## 8.4 对 Studio 内嵌 Drive 的最终建议

建议定位成：

> **Studio 的“最近录制切换器”**

而不是把完整 Drive 全量搬进 Studio。

这能让用户理解它的作用是：

- 切换当前编辑对象；
- 快速回到其他录制；
- 必要时删除或查看基础信息。

---

## 9. 推荐验收指标

上线后建议至少跟踪以下指标：

1. Action 菜单三入口点击占比
2. Action → Record 的点击转化率
3. Action → Studio 的点击占比
4. Studio 空状态出现率
5. 空状态下“开始录制”按钮点击率
6. Studio 内切换录制次数
7. Drive / Studio 页面访问用户数
8. 安装后 1 天 / 7 天卸载率变化

---

## 10. 最终结论

## 10.1 是否值得做

**值得做，而且应尽快做。**

## 10.2 是否能降低卸载率

**能。**

但更准确地说，它降低的是以下几类卸载：

- 因入口与预期不匹配造成的卸载；
- 因找不到历史视频造成的卸载；
- 因 Studio 无法直接进入或无解释空状态造成的卸载。

## 10.3 最终建议

建议按照以下策略推进：

1. **先重构 Action 入口，建立 Record / Drive / Studio 三分流；**
2. **让 Studio 成为可直接进入的工作台，而不是只接受录制完成后的跳转页；**
3. **把 Studio 空状态做成明确的“下一步引导页”；**
4. **再把 Drive 以抽屉 / 弹层形式并入 Studio，形成完整闭环。**

如果只能先做一部分，我的推荐顺序是：

> **Action 菜单 > Studio 空状态 > Studio 默认打开最近录制 > Studio 内嵌 Drive**

这是当前工程条件下，最有可能以较小改造换取较明显留存改善的一组方案。
