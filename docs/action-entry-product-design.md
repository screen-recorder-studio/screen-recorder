# Action 入口重构与 Studio 工作台产品设计文档

## 1. 文档信息

- 项目：Screen Recorder Studio
- 文档类型：产品设计文档
- 设计目标：
  1. 让 action 成为统一的任务入口，而不是单一录制入口
  2. 缩短用户找到历史录制与进入 Studio 的路径
  3. 提升 Studio 的一级入口能力与空状态可用性
  4. 在 Studio 内提供轻量级录制切换能力

---

## 2. 设计背景

当前 action 点击后直接打开录制控制页，适合“马上开始录制”的场景，但存在以下产品缺口：

1. 历史录制不在一级入口暴露；
2. Studio 不是 action 一级入口；
3. 用户如果想继续编辑上一次录制，需要先进入控制页，再跳 Drive，再进 Studio；
4. Studio 缺少系统性的无视频引导，不适合直接作为 action 入口。

本设计的目标不是推翻现有录制链路，而是在尽量复用现有 Control / Drive / Studio 能力的前提下，重构 action 的信息架构。

---

## 3. 产品目标

## 3.1 目标

### 目标一：降低入口认知门槛

让用户点击 action 后，能够立即理解这个产品至少能做三件事：

- 录制
- 管理录制
- 编辑导出

### 目标二：缩短高频任务路径

将以下任务压缩到一级入口：

- 开始一次新的录制
- 找到历史录制
- 打开最近一次录制继续编辑

### 目标三：让 Studio 成为真正的工作台

Studio 不仅承接“录完后自动打开”，也承接“主动进入继续处理”的场景。

### 目标四：保证空状态自解释

任何“没有视频”的场景都不能变成空白页或失败体验。

---

## 3.2 非目标

本次设计不解决以下问题：

1. 录制稳定性、权限异常、音视频兼容问题；
2. 导出性能与编码优化；
3. Drive 的完整信息架构重构；
4. Studio 的编辑能力大改版；
5. 新增云同步、账号体系、跨设备资产能力。

---

## 4. 目标用户与核心场景

## 4.1 用户类型

### A. 首次用户

目标：

- 尝试快速开始录制；
- 理解产品有哪些能力；
- 录完后看到结果在哪里。

### B. 回访轻度用户

目标：

- 找到最近录过的视频；
- 简单预览或导出；
- 再录一次。

### C. 回访重度用户

目标：

- 在多个录制之间切换；
- 调整背景、边距、样式；
- 反复导出和管理内容。

---

## 4.2 核心场景

### 场景一：我要马上开始录制

点击 action → 选择“录制” → 打开 Control → 开始录制

### 场景二：我要找上次录的视频

点击 action → 选择“Drive” → 直接看到录制列表

### 场景三：我要继续处理最近的视频

点击 action → 选择“Studio” → 自动打开最近一条可用录制

### 场景四：我还没有任何视频

点击 action → 选择“Studio” → 看到空状态引导 → 点击“开始录制”

### 场景五：我在 Studio 想切换到另一条录制

点击 Studio 顶部“Drive”按钮 → 右侧抽屉 / 弹层出现 → 选择另一条录制 → 当前 Studio 切换内容

---

## 5. 信息架构设计

## 5.1 Action 入口新结构

Action 不再默认直接进入控制页，而是进入一个轻量级 launcher。

### 一级入口

1. **录制**
2. **Drive**
3. **Studio**

### 设计原则

- 只做任务分流，不做复杂操作；
- 视觉上突出“录制”，但不隐藏 Drive / Studio；
- 一屏内完成理解与选择；
- 不承担持续状态面板职责。

---

## 5.2 页面职责边界

### Action Launcher

负责：

- 告知三条主路径；
- 快速跳转到对应页面；
- 可选展示“最近录制状态摘要”。

不负责：

- 完整录制控制；
- 复杂列表浏览；
- 视频编辑。

### Control

负责：

- 录制模式选择；
- 开始 / 暂停 / 停止；
- 录制过程状态反馈。

### Drive

负责：

- 浏览录制资产；
- 删除与管理；
- 进入 Studio。

### Studio

负责：

- 预览与编辑当前视频；
- 导出；
- 切换最近录制；
- 在无视频时承担引导职责。

---

## 6. 核心交互设计

## 6.1 Action Launcher 交互设计

### 入口形式

建议采用 **Chrome action popup** 作为 launcher。

### 视觉结构

```text
┌──────────────────────────┐
│ Screen Recorder Studio   │
│ Record, manage, export   │
├──────────────────────────┤
│ [ 录制 ]                 │
│ 开始新的录屏             │
├──────────────────────────┤
│ [ Drive ]                │
│ 查看历史录制             │
├──────────────────────────┤
│ [ Studio ]               │
│ 继续编辑最近录制         │
├──────────────────────────┤
│ 最近录制：2小时前        │
└──────────────────────────┘
```

### 行为定义

#### 点击“录制”

- 关闭 launcher；
- 打开或聚焦现有 `control.html` 窗口；
- 保持当前 Control 页逻辑不变。

#### 点击“Drive”

- 关闭 launcher；
- 打开 `drive.html`。

#### 点击“Studio”

- 关闭 launcher；
- 尝试查找最近一条可用录制；
- 若存在，则打开 `studio.html?id=<latestValidId>`；
- 若不存在，则打开 `studio.html`，进入 Studio 空状态。

### 设计说明

这里的 launcher 是“跳转页”，不是录制主界面。  
这样能避免 popup 关闭即丢失交互状态的问题，也不影响现有 Control 页作为录制控制台的价值。

---

## 6.2 Studio 默认打开逻辑

## 6.2.1 入口规则

Studio 存在两种打开方式：

1. **带 `id` 参数打开**
   - 直接加载指定录制；
2. **不带 `id` 参数打开**
   - 自动加载最近一条可用录制；
   - 如果没有可用录制，则展示空状态页。

## 6.2.2 “最近一条可用录制”的定义

按照 Drive 默认排序规则处理：

1. 遍历 OPFS 录制目录；
2. 读取 `meta.json`；
3. 按 `createdAt` 倒序排序；
4. 从前往后查找第一条“可正常打开”的录制；
5. 若某条缺少关键文件或读取失败，则跳过；
6. 全部不可用时，进入空状态。

### 不建议采用的定义

- 按目录扫描顺序第一条；
- 按文件名字典序第一条；
- 只要最新，不校验可用性。

---

## 6.3 Studio 空状态设计

## 6.3.1 触发场景

以下任一情况进入 Studio 空状态：

1. 用户主动从 action 点击 Studio，但没有任何录制；
2. 最近录制不存在；
3. 最近录制损坏或不可用；
4. 用户在 Studio 中删除了当前最后一条录制；
5. OPFS 不可访问。

## 6.3.2 空状态目标

空状态必须回答四个问题：

1. 这里是什么页面？
2. 为什么现在没有内容？
3. 我下一步该做什么？
4. 录完之后会怎样？

## 6.3.3 UI 文案建议

### 标题

**还没有可编辑的视频**

### 说明文案

完成录制后，视频会自动出现在这里。  
你可以在 Studio 中预览、裁剪、美化并导出视频。

### 主按钮

**开始录制**

行为：

- 打开 Control 页面；
- 若当前已有 Control 窗口，则聚焦。

### 次按钮

**打开 Drive**

行为：

- 打开 Drive 页面；
- 供用户确认历史数据或手动管理。

### 辅助说明

- 支持预览
- 支持背景与样式调整
- 支持导出成最终视频

---

## 6.4 Studio 内嵌 Drive 设计

## 6.4.1 设计目标

让用户在不离开 Studio 的情况下：

- 查看最近录制；
- 切换当前编辑对象；
- 删除某条录制；
- 快速回到其他内容。

## 6.4.2 交互形态

建议使用：

> **右侧抽屉或覆盖式面板**

而不是：

- 永久并排展示完整 Drive；
- 新开一个标签页；
- 使用浏览器原生弹窗。

### 推荐原因

1. 不压缩主预览画布；
2. 不打断当前编辑上下文；
3. 符合“偶尔切换录制”的使用频次；
4. 容易复用现有 Drive 列表组件。

## 6.4.3 打开方式

Studio 顶部现有 “Drive” 按钮，升级为：

- 默认点击：在当前页面打开 Drive 抽屉；
- 可选二级入口：保留“在新页打开 Drive”能力，但不作为主路径。

## 6.4.4 面板内容

建议面板只保留“最近录制切换”相关能力：

- 录制列表
- 缩略图
- 标题 / 时间 / 时长
- 当前选中态
- 删除
- 打开到当前 Studio

不建议在面板中承载：

- 过重的批量管理；
- 复杂筛选排序；
- 多列布局。

---

## 7. 关键流程设计

## 7.1 流程 A：从 action 开始录制

```text
点击 action
→ 打开 Launcher
→ 点击“录制”
→ 打开 / 聚焦 Control
→ 选择模式并开始录制
→ 录制结束
→ 自动打开 Studio(id)
```

## 7.2 流程 B：从 action 进入最近录制

```text
点击 action
→ 打开 Launcher
→ 点击“Studio”
→ 查找最近一条可用录制
→ 打开 Studio(id)
```

## 7.3 流程 C：没有录制时进入 Studio

```text
点击 action
→ 打开 Launcher
→ 点击“Studio”
→ 未找到可用录制
→ 打开 Studio 空状态
→ 点击“开始录制”
→ 打开 / 聚焦 Control
```

## 7.4 流程 D：在 Studio 内切换录制

```text
进入 Studio
→ 点击“Drive”
→ 打开侧边抽屉
→ 选择另一条录制
→ 当前 Studio 切换到新录制
→ 关闭抽屉
```

## 8. 页面与组件设计建议

## 8.1 Action Launcher 页面建议

### 页面定位

可复用现有 popup 路由作为 launcher，也可新增专用 launcher 路由。  
从产品清晰度角度，更推荐把 action popup 变成明确的 launcher，而不是继续让它承担完整录制控制。

### 推荐布局

- 顶部：品牌 + 简短描述
- 中部：三张主操作卡片
- 底部：最近录制摘要 / 最近打开时间 / 版本号

### 卡片内容

#### 录制卡

- 图标：录制 / 播放
- 标题：录制
- 描述：开始新的屏幕录制
- 主色强调

#### Drive 卡

- 图标：硬盘 / 文件
- 标题：Drive
- 描述：查看和管理历史录制

#### Studio 卡

- 图标：视频 / 编辑
- 标题：Studio
- 描述：继续编辑最近录制

---

## 8.2 Studio 页面建议

### 默认态

- 有视频：正常进入编辑态
- 无视频：进入空状态

### 空状态布局建议

```text
┌────────────────────────────────┐
│ Studio                         │
├────────────────────────────────┤
│ [插画 / 图标]                  │
│ 还没有可编辑的视频             │
│ 录制完成后，视频会自动出现在这里 │
│                                │
│ [开始录制]  [打开 Drive]       │
│                                │
│ 可在此预览 / 裁剪 / 导出        │
└────────────────────────────────┘
```

### 编辑态中 Drive 入口

- 顶部按钮文案可从“Drive”微调为“最近录制”；
- 点击后打开抽屉；
- 选中当前视频应有高亮态。

## 9. 状态与边界条件设计

## 9.1 无视频

行为：

- 打开 Studio 空状态；
- 不报错，不显示空白画布；
- 主动给出录制路径。

## 9.2 当前视频损坏

行为：

- 若通过 action 打开 Studio，自动跳过该条，尝试下一条可用录制；
- 若全部不可用，进入空状态；
- 给出轻量提示：“最近录制不可读取，已为你打开空白工作台”。

## 9.3 当前正在录制

行为建议：

- Action Launcher 中点击“录制”时，优先聚焦当前 Control；
- 点击“Studio”时，可正常打开最近一次已完成录制；
- 不应误导用户去编辑“尚未完成写入”的当前录制。

## 9.4 Studio 中删除最后一条视频

行为：

- 删除成功后，如果列表为空，立即切回 Studio 空状态；
- 不保留无意义的旧画面。

## 9.5 OPFS 不可访问

行为：

- 显示明确异常提示；
- 仍保留“开始录制”按钮；
- 不让用户停留在无法理解的状态。

## 10. 文案策略

## 10.1 Action Launcher 文案原则

- 用用户目标命名，而不是内部结构命名；
- “录制”优先使用动词；
- “Drive / Studio”可保留品牌术语，但需配一句解释。

### 建议示例

- 录制：开始新的屏幕录制
- Drive：查看和管理历史录制
- Studio：继续编辑最近录制

## 10.2 Studio 空状态文案原则

- 不让用户产生“页面坏了”的误解；
- 明确说明内容出现条件；
- 明确说明下一步；
- 强化 Studio 的价值认知。

## 12. 实施优先级建议

## Phase 1：入口与空状态成型

目标：

- 先把“能不能直达正确页面”解决。

范围：

1. Action Launcher 三入口
2. Studio 无 `id` 时自动查找最近可用录制
3. Studio 空状态

验收标准：

- 用户可从 action 一步进入 Drive；
- 用户可从 action 一步进入最近录制；
- 无录制时 Studio 不出现空白页。

## Phase 2：Studio 内录制切换

目标：

- 建立编辑工作台闭环。

范围：

1. Studio 内嵌 Drive 抽屉
2. 当前录制切换
3. 列表复用与轻量管理

验收标准：

- 用户不离开 Studio 即可切换录制；
- 切换后当前编辑内容立即更新；
- 空状态与删除逻辑连贯。

## Phase 3：体验增强

范围：

1. Action 展示最近录制摘要
2. Studio 中保留最近打开记录
3. 空状态增加帮助内容

---

## 13. 成功标准

本方案上线后，满足以下结果即可视为成功：

1. Action 入口不再只服务录制意图；
2. Drive 和 Studio 成为可感知的一线能力；
3. 用户从 action 到历史视频的路径显著缩短；
4. Studio 在无视频情况下仍具备明确引导；
5. 用户在 Studio 内切换录制不再需要离开当前页面。

---

## 14. 最终设计结论

本次产品设计建议将 Screen Recorder Studio 的 action 入口重新定义为：

> **录制工作台总入口**

核心方案如下：

1. **Action 使用轻量 launcher，提供“录制 / Drive / Studio”三入口；**
2. **Studio 支持无 `id` 直接打开最近一条可用录制；**
3. **若没有录制，则展示具备明确 CTA 的 Studio 空状态；**
4. **Studio 内通过抽屉式 Drive 提供录制切换能力，形成编辑闭环。**

这个方案兼顾了：

- 首次用户的理解成本；
- 回访用户的任务效率；
- 现有工程结构的可复用性；
- 后续演进为完整工作台的空间。

从产品视角看，这是一次**入口架构升级**，不只是一次按钮改动。
