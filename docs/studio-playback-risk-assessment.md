# Studio 播放优化风险评估

> 基于评估报告：`docs/studio-playback-evaluation.md`  
> 基于技术方案：`docs/studio-playback-technical-solution.md`

---

## 一、总体风险评级：🟢 低风险

本次修复涉及 3 个独立的、局部的代码变更，不改变架构、不新增依赖、不修改数据格式。每项修复均有明确的回退策略。

---

## 二、逐项风险分析

### P0：播放循环帧时间累积修复

| 维度 | 评估 | 说明 |
|------|------|------|
| 影响范围 | 仅 `composite-worker/index.ts` 的 `playFrame()` 函数 | 不影响解码、合成、缓冲等其他逻辑 |
| 兼容性 | 无 API 变更，消息格式不变 | 主线程无需任何适配 |
| 边界情况 | 安全阀机制防止暂停恢复后快进 | `if (now - lastFrameTime > frameInterval * 2)` 重新同步 |
| 回退策略 | 将 `lastFrameTime += frameInterval` 改回 `lastFrameTime = now` | 1 行代码还原 |
| 风险等级 | 🟢 **极低** | 标准的时间累积方案，广泛用于游戏引擎和动画框架 |

**潜在边缘场景**：
- 浏览器标签页被后台挂起时，`performance.now()` 可能出现大跳跃 → 安全阀会触发重新同步，已覆盖。
- 帧率非整数时（如 29.97fps），`frameInterval` 为浮点数 → 累积误差在可接受范围内（每小时 < 1ms）。

### P1：窗口切换 rAF 优化

| 维度 | 评估 | 说明 |
|------|------|------|
| 影响范围 | 仅 `VideoPreviewComposite.svelte` 的 `shouldContinuePlayback` 处理段 | 不影响 Seek、Trim、预览等功能 |
| 兼容性 | Worker 消息格式和顺序不变 | 'seek' 和 'play' 消息内容相同 |
| 边界情况 | Worker FIFO 消息队列保证处理顺序 | HTML 规范保证同一事件循环中的 postMessage 按顺序到达 |
| 回退策略 | 重新嵌套一层 rAF | 2 行代码还原 |
| 风险等级 | 🟢 **低** | 减少异步层级，逻辑更简单 |

**潜在边缘场景**：
- 如果 `seekToFrame()` 内部的 `postMessage` 在极端情况下抛出异常 → 后续 `play()` 不会执行，播放会停止但不会崩溃。用户可手动点击播放恢复。这与修改前的行为一致（第一个 rAF 中抛异常同样会阻止第二个 rAF 执行）。

### P2：时间戳精度优化

| 维度 | 评估 | 说明 |
|------|------|------|
| 影响范围 | `timestampToMs()` 函数 + `+page.svelte` 窗口时间计算 | 影响所有使用该函数的时间戳转换 |
| 兼容性 | 返回值类型不变（整数毫秒） | 调用方无需适配 |
| 边界情况 | `Math.round()` 在 `.5` 时向上取整 | 与 `Math.floor()` 的最大差异为 1ms，在帧间距 33ms 的上下文中可忽略 |
| 回退策略 | `Math.round` → `Math.floor` | 2 处代码还原 |
| 风险等级 | 🟢 **极低** | 标准精度优化 |

**潜在边缘场景**：
- 二分查找 `idxByRelativeTimeMs()` 使用 `<=` 比较，`Math.round()` 可能导致多返回或少返回 1 帧 → 影响范围 ≤ 1 帧（33ms），用户不可感知。

## 三、组合风险评估

| 场景 | 风险 | 说明 |
|------|------|------|
| 三项修复同时生效 | 🟢 无耦合 | 三处修改分别在不同线程（Worker/主线程）和不同代码路径 |
| 与 Trim 功能交互 | 🟢 无影响 | Trim 的帧边界检查在 `displayFrame()` 中，不受播放循环时间管理影响 |
| 与 Zoom 功能交互 | 🟢 无影响 | Zoom 渲染在 `renderCompositeFrame()` 内部，不依赖帧时间变量 |
| 与导出功能交互 | 🟢 无影响 | 导出使用独立的 Worker 和渲染管道，不共享播放循环代码 |
| 与预取机制交互 | 🟢 无影响 | 预取在 `+page.svelte` 中独立管理，不受播放循环时间变量影响 |

---

## 四、结论

所有修复项均为 **🟢 低风险**，建议直接实施。每项修复都是独立的、可单独回退的局部变更，不涉及架构调整或数据格式变更。
